## 🎯 Приоритет 1: По ТЗ для MVP+ (критерии приёмки и «желательно»)

### 1. Экспорт данных
**ТЗ:** «Экспорт: Excel / PDF-отчёт "ESG Data Landscape 202X"», «Работает экспорт в Excel».

- [ ] **Экспорт в Excel (.xlsx)** — выгрузка метрик/источников/подразделений с учётом текущих фильтров.
- [ ] **Экспорт в CSV** — для быстрой выгрузки и интеграции.
- [ ] **PDF-отчёт** — сводный отчёт «ESG Data Landscape 202X» (таблица метрик + статусы + ответственные).

*Реализация:* бэкенд — маршрут `GET /api/export/metrics?format=xlsx|csv` (библиотеки `exceljs`, `csv-stringify`), для PDF — `puppeteer` или `pdfkit` / фронтенд — кнопки «Экспорт» на страницах списков и дашборда.

### 2. Страница «О системе» + глоссарий + методология
**ТЗ:** «Страница "О системе" + глоссарий + методология заполнения».

- [ ] **Страница «О системе»** — описание цели системы, контакты поддержки, версия.
- [ ] **Глоссарий** — определения: E/S/G, GRI, STARS, Scope 1/2/3, data owner, data steward и т.д.
- [ ] **Методология заполнения** — краткая инструкция, как заполнять метрики, источники и связи (кто что вносит, как трактовать статусы и качество).

*Реализация:* новые маршруты `/about`, `/glossary`, `/methodology` и одноимённые Vue-страницы (контент можно хранить в Markdown или JSON).

### 3. Расширение модели метрик (стандарты и детализация)
**ТЗ:** «Соответствие стандартам (GRI, SASB, TCFD, SDGs, STARS)», «Scope 1/2/3 (для E), подкатегории для S и G».

- [ ] В модели **Metric** добавить:
  - `scope` (для E): `SCOPE_1` | `SCOPE_2` | `SCOPE_3` | null.
  - `standards` — массив или строка стандартов: GRI, SASB, TCFD, SDG, STARS (можно JSON или отдельная таблица связей).
  - `subcategory` — подкатегория внутри E/S/G (опционально).
- [ ] В UI — выбор scope при создании/редактировании метрики, мультивыбор стандартов.

### 4. Расширение подразделений (контакты и конфиденциальность)
**ТЗ:** «Ответственный за сбор (data steward)», «Контакты (email, телефон, Teams/Telegram)», «Уровень доступа / конфиденциальность».

- [ ] В модели **Department** (или в связке с метрикой) добавить:
  - `dataSteward` — ФИО/роль ответственного за сбор.
  - `phone` — телефон.
  - `contactChannel` — Teams / Telegram / другой мессенджер (строка или enum).
  - `accessLevel` или `confidentiality`: публичные / внутренние / чувствительные (enum).
- [ ] Отображать в карточке подразделения и в деталях метрики.

### 5. Расширение источников данных
**ТЗ:** «Тип источника: внутренний отчёт / внешний поставщик / счётчик / опрос / ERP / ручной ввод», «Конкретные системы / файлы».

- [ ] В **DataSource** добавить:
  - `sourceKind`: внутренний отчёт | внешний поставщик | счётчик | опрос | ERP | ручной ввод (enum или строка).
  - `systemOrFile` — текст (конкретная система/файл, например «Excel от энергетиков», «1С», «Google Forms»).
- [ ] В UI — выбор типа и поле для уточнения системы/файла.

### 6. Heatmap / таблица gaps
**ТЗ:** «Heatmap / таблица gaps: где самые большие пробелы в данных».

- [ ] **Страница «Gaps / Пробелы»**:
  - Таблица или матрица: оси — блоки E/S/G (или подкатегории) × подразделения (или источники).
  - Цвет/индикатор: нет данных, частично, собирается (по статусу и качеству из `MetricLink`).
- [ ] Фильтры по стандарту, по подразделению.
- [ ] Опционально: экспорт этой таблицы в Excel/PDF.

*Реализация:* бэкенд — агрегирующий эндпоинт `GET /api/metrics/gaps` или расчёт на фронте по существующим API; фронт — страница с таблицей/heatmap (например, CSS или лёгкая библиотека визуализации).

### 7. Журнал изменений (аудит)
**ТЗ:** «Журнал изменений (кто и когда добавил/изменил запись)».

- [ ] **Аудит-логирование**:
  - Модель `AuditLog`: userId, entityType (metric/source/department/storage/link), entityId, action (create/update/delete), oldValues/newValues (JSON), timestamp.
  - При каждом создании/обновлении/удалении — запись в `AuditLog`.
- [ ] **UI**: страница «Журнал изменений» (только для admin) с фильтрами по сущности, пользователю, дате; опционально — блок «Последние изменения» на дашборде.

## 📋 Приоритет 3: Дополнительно по ТЗ и масштабированию

- **Версионность и актуальность:** уже есть `lastUpdate` в `MetricLink`; можно добавить явное «дата последнего обновления записи» на уровне метрики/источника и отображать в списках.
- **Оценка качества по шкале 1–5 или цвет:** сейчас `qualityScore` 0–100; можно добавить отображение цветом (красный/жёлтый/зелёный) и легенду в глоссарии/методологии.
- **Заполнение 30–50+ записей:** проверить seed и при необходимости добавить больше тестовых метрик/источников/связей под реальные сценарии вуза.

## Рекомендуемый порядок внедрения

1. **Экспорт в Excel/CSV** — быстро закрывает критерий приёмки и запросы пользователей.
2. **Страницы «О системе», глоссарий, методология** — требуются по ТЗ, реализуются без смены схемы БД.
3. **Расширение схемы** (scope, standards, контакты, тип источника) — одна миграция Prisma, затем обновление API и форм.
4. **Heatmap/Gaps** — после расширения метрик даёт максимальную пользу для «карты данных».
5. **Журнал изменений** — модель + middleware/хуки при сохранении, затем страница для админов.

Файл можно использовать как дополнение к `ROADMAP.md`: в ROADMAP — общие фазы продукта, здесь — явная привязка к ТЗ и приоритеты для MVP+.
